# Rule: Research Code Reproducibility

Standards for ensuring that computational research is fully reproducible —
another researcher can obtain the same results from the same inputs.

## The Four Requirements of Reproducibility

1. **Same code** — version-controlled, tagged, citable
2. **Same data** — archived, DOI-referenced, or reconstructible
3. **Same environment** — locked dependency versions, containerised
4. **Same randomness** — fixed seeds, logged

## Code and Version Control

- All analysis code lives in a git repository
- Each result in the paper/report corresponds to a tagged commit or release
- No analysis code in ad-hoc scripts outside the repository
- Data-cleaning steps are code, not manual spreadsheet edits

## Dependency Management

```
# REQUIRED: one of these must be present
requirements.txt      # pip: exact pinned versions (pip freeze > requirements.txt)
environment.yml       # conda: pinned versions including build strings
pyproject.toml        # poetry/hatch with locked versions (poetry.lock present)
renv.lock             # R: renv lock file
```

- Docker image preferred for full environment isolation
- Dockerfile must use a pinned base image tag, not `latest`
- Include `README` section explaining how to recreate the environment

## Data Provenance

- Every dataset must have a data card specifying:
  - Source URL and access date
  - License
  - Version or hash
  - Any preprocessing steps
- Raw data is read-only; processed data is regenerated by running the pipeline
- Use DVC, Git LFS, or Zenodo for large files

## Random Seeds

```python
# Required at top of EVERY script/notebook that uses randomness
SEED = 42
import random; random.seed(SEED)
import numpy as np; np.random.seed(SEED)
# Add framework-specific seeds as needed
```

## Pipeline Structure

```
project/
├── data/
│   ├── raw/          # read-only, never modified
│   └── processed/    # generated by pipeline
├── src/              # reusable code
├── notebooks/        # analysis notebooks (numbered, runnable top-to-bottom)
├── scripts/          # standalone scripts
├── figures/          # generated by scripts; not committed (in .gitignore)
├── results/          # tables, metrics (committed only if small)
└── Makefile          # or DVC pipeline: single command to reproduce everything
```

## The Makefile / Pipeline

A single command must reproduce all results:
```makefile
all: results/main_table.csv figures/fig1.png figures/fig2.png

data/processed/clean.csv: data/raw/survey.csv src/preprocess.py
    python src/preprocess.py

results/main_table.csv: data/processed/clean.csv src/analysis.py
    python src/analysis.py
```

## Review Checklist

- [ ] All code in a version-controlled repository
- [ ] Dependency lock file present (`requirements.txt`, `poetry.lock`, etc.)
- [ ] Random seeds fixed and documented
- [ ] Raw data is read-only
- [ ] Data provenance documented (source, license, version)
- [ ] Pipeline reproducible with a single command (`make`, `dvc repro`)
- [ ] README has "How to reproduce" section
- [ ] No absolute paths anywhere
- [ ] Results tied to a specific commit/tag
